**规范驱动开发的核心模板和指南**

| 文档 | 内容概要 | 使用场景 |
|------|----------|----------|
| [SDD 规范模板](./02-SDD软件设计规范/SDD规范模板.md) | 完整的功能设计规范模板 + 设计模式集成指南 | 任何新功能开发前必读 |

**新增内容** (基于 @rules/ 目录规范):
- **设计模式集成**: Builder、Factory、Strategy、Observer等模式应用指南
- **AI 提示模板**: 如何描述设计模式让 AI 正确生成代码
- **架构集成**: 分层架构中的模式应用
- **Spring Boot 集成**: 使用注解实现各种设计模式

**文档结构**:
1. 文档头部 (功能名称、编号、状态等)
2. **设计模式选择** (基于 @rules/02-design/patterns.mdc)
3. 概述 (目标、背景、范围)
4. 需求详细说明 (功能/非功能需求)
5. 技术设计 (架构、模块、接口)
6. 测试策略 (单元/集成/性能测试)
7. 安全考虑
8. 性能优化

**使用流程**:
```
需求分析 → 选择设计模式 → 编写 SDD → 团队评审 → AI 生成代码 → 实现验证
```

**设计模式应用示例**:
```markdown
### 2.1.1 用户认证功能
**设计模式**: Strategy + Factory
- 认证策略接口：AuthenticationStrategy
- 具体策略：PasswordAuthStrategy, SmsAuthStrategy
- 策略工厂：AuthenticationStrategyFactory

**AI 提示**: "使用 Strategy 模式实现认证功能，支持多种认证方式"
```

- AAA 测试模式 (Arrange-Act-Assert)
- Mock 和 Stub 使用
- 实际项目示例

**关键实践**:
```typescript
// 1. 先写失败的测试
it('should calculate total price', () => {
  const cart = new ShoppingCart();
  cart.addItem({ price: 10, quantity: 2 });
  expect(cart.totalPrice).toBe(20);
});

// 2. 编写最小实现代码
class ShoppingCart {
  constructor(public totalPrice = 0) {}
  addItem(item: any) {
    this.totalPrice += item.price * item.quantity;
  }
}

// 3. 重构优化
```

### 4. 架构设计规范
**架构决策和技术选型指南**

| 文档 | 内容概要 | 适用角色 |
|------|----------|----------|
| [架构设计原则](./04-架构设计规范/架构设计原则.md) | SOLID 原则、设计模式、微服务架构 + @rules/ 项目结构规范 | 架构师、高级开发 |

**新增内容** (基于 @rules/ 目录规范):
- **强制项目结构**: 详细的包结构和目录要求
- **技术栈规范**: Spring Boot + MyBatis + Gradle 完整要求
- **设计模式规范**: L2层级的设计模式应用标准
- **编码规范**: 命名、注释、测试等具体要求

**主要内容**:
- **SOLID 原则**: 详细示例和实践指导
  - SRP: 单一职责原则
  - OCP: 开闭原则
  - LSP: 里氏替换原则
  - ISP: 接口隔离原则
  - DIP: 依赖倒置原则
- **模块设计**: 高内聚、低耦合的实现
- **架构模式**: MVC、六边形、CQRS、事件溯源
- **微服务设计**: 服务拆分、通信方式、数据一致性
- **性能优化**: 缓存策略、数据库优化
- **@rules/ 集成**: 项目结构、技术栈、设计模式、编码规范

**架构决策记录 (ADR)**:
```markdown
# ADR-001: 使用微服务架构

## 背景
单体应用规模扩大，团队增长...

## 决策
拆分为微服务：用户、订单、商品服务

## 后果
积极: 独立部署、技术异构
消极: 复杂性增加
```

### 5. 技术栈规范
**核心技术选型和工具使用**

| 文档 | 内容概要 | 技术维度 |
|------|----------|----------|
| [技术栈规范](./05-技术栈规范/技术栈规范.md) | Spring Boot 强制规范 + 前端/后端/数据库/工具链 | 全栈技术选型 |

**新增内容** (基于 @rules/ 目录规范):
- **Spring Boot 强制规范**: Gradle构建、MyBatis注解模式、统一响应模型
- **代码结构规范**: 详细的包结构和类命名要求
- **设计模式应用**: 各层架构中的模式使用指南
- **代码规范集成**: 控制器、服务层、数据访问的具体实现规范

**Spring Boot 技术栈 (强制)**:
- **构建工具**: Gradle ≥ 8.14 (禁止Maven)
- **核心框架**: Spring Boot + Spring MVC + Spring Security
- **数据访问**: MyBatis注解模式 + MySQL 8.0+
- **API文档**: Springdoc OpenAPI (Swagger UI)
- **测试框架**: JUnit 5 + Mockito + Spring Boot Test

**前端技术栈**:
- **框架**: React 18+ / Next.js 14
- **语言**: TypeScript 5.3+
- **UI**: Ant Design 5.x / Tailwind CSS
- **构建**: Vite 5.x
- **测试**: Jest + React Testing Library

**后端技术栈**:
- **Node.js**: NestJS 10+ (企业级框架)
- **Python**: FastAPI (高性能)
- **数据库**: PostgreSQL 15+ (首选)
- **缓存**: Redis 7+
- **消息队列**: RabbitMQ / Kafka

**基础设施**:
- **容器化**: Docker
- **CI/CD**: GitHub Actions / GitLab CI
- **监控**: Prometheus + Grafana
- **日志**: ELK Stack

**决策矩阵**:
| 技术 | 适用场景 | 优势 |
|------|----------|------|
| Next.js | SSR/全栈应用 | SEO友好、性能好 |
| NestJS | 企业级API | TypeScript、IOC容器 |
| PostgreSQL | 关系型数据 | 功能丰富、稳定 |
| Redis | 缓存/队列 | 高性能、多功能 |

5. ✅ 了解相关技术栈规范
6. ✅ 设置开发环境

#### 编码过程中（必须遵循）
1. ✅ 遵循 TDD 流程 (红-绿-重构)
2. ✅ 使用强制目录结构（参考架构设计原则）
3. ✅ 应用设计模式（参考SDD模板中的模式指南）
4. ✅ 遵守代码规范（控制器、服务层、数据访问规范）
5. ✅ 编写单元测试（JUnit 5 + Mockito）
6. ✅ 使用统一响应模型（CommonResponse）
7. ✅ 遵循异常处理规范（@ControllerAdvice）
8. ✅ 符合日志规范（SLF4J + AOP）

#### 代码提交前（强制检查）

2. ✅ 代码覆盖率 ≥ 80%
3. ✅ 符合 Code Review 标准
